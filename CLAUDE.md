# DSG — Design Systems Guide

## ⚠️ Перед коммитом и заливкой — проверь контекст (обязательно, во всех чатах)

Над проектом параллельно работают несколько чатов/процессов. **Никогда не коммить и не деплой вслепую.** Перед любым коммитом и пушем обязательно:

1. `git fetch origin` и сверь локальную ветку с `origin/main` (`git rev-list --left-right --count origin/main...HEAD`).
2. Посмотри свежие коммиты (`git log --oneline --all -8`) — что наделали другие чаты/процессы, не пересобирался ли `digests.json`/каталог, нет ли встречных правок тех же файлов.
3. Проверь рабочее дерево (`git status`) — нет ли чужих незакоммиченных изменений.
4. Если трогаешь данные дайджестов — прогони скан аномалий описаний (заглушки «Материал разбирает тему…», склейки таксономии, обрезанные начала, английский/пустые) **до** и **после** правки.
5. Только убедившись, что ничего не сломаешь и не перезатрёшь чужую работу — коммить и деплой.

При расхождении (origin впереди) — сначала `git pull --rebase`, разрулить, и только потом пушить. Цель: понимать контекст и не ломать то, что катит другой процесс.

## Workflow: изменения → тест → прод

После **каждого** изменения кода обязательная последовательность:

1. **Коммит** — зафиксировать изменения в текущей ветке (не в main)
2. **Деплой на тест** — запушить в non-main ветку → GitHub Pages (`lorrrem-zlnkh.github.io/DSG/`)
3. **Проверка** — убедиться что сайт работает, изменения видны, ничего не сломано
4. **Деплой на прод** — только после успешной проверки: смержить/запушить в `main` → FTP → `dsg.lorrrem.ru`

Никогда не пушить напрямую в `main` без предварительной проверки на тестовой ветке.
